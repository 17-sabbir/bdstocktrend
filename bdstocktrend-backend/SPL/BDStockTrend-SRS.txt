1. Introduction
The introduction of the Software Requirements Specification (SRS) provides an overview of the entire SRS with purpose, scope, abbreviations, references and overview of the SRS. The aim of this document is to gather and analyze and give an in-depth insight of the complete “BDStockTrend Application”. By presenting the problem statement in detail, the purpose of this document is to collect, evaluate, and provide a deeper understanding of the whole "BDStockTrend" application. Nevertheless, it also concentrates on the capabilities required by stakeholders and their needs. The detailed requirements of the “BDStockTrend” are provided in this document.

1.1 Problem Statement 
There is a lack of mobile applications that help investors to have forecast about the stock trends of Bangladeshi companies. The nature of stock prices can lead any investors in enormous loss for wrong or immature prediction of stock trends.

Therefore, a mobile application with reasonably accurate forecasting ability using state-of-the-art artificial intelligence technologies is about time. Bangladeshi investors rely on heuristic techniques for stock market predictions, aiming for risk-free profits. However, the volatile nature of stock prices can lead to significant losses due to inaccurate predictions. This motivates to develop a mobile application with reasonably accurate forecasting ability using state-of-the-art artificial intelligence technologies.

1.2 Purpose
The purpose of this document is to collect and analyze all assorted ideas that have come up to define the system and all the various ideas that have emerged in order to develop the program. Also, to get a better idea of ??the project, describe the ideas that may be developed later, and write down ideas that are being considered but which can be eliminated as the system progresses, we will predict and plan how we expect this plan to be implemented.
In short, the purpose of this SRS document is to provide a comprehensive overview of our software product, including its specifications and objectives. This document describes the user interface, hardware, and software requirements, as well as the project's target audience. Specifies how the system and its functions are viewed by administrators, teachers, and students. However, it assists any designer or developer in software delivery processes (SDLC).

1.3 Project Scope
The documentation of a project's scope is called a scope statement or terms of reference. This project traverses a lot of areas ranging from business concept to computing field and is required to perform several researches to be able to achieve the project objectives. The following are the scopes of work:

A mobile application for stock traders of Bangladesh. In the application user will able to register and login to use the free service of the application. For premium service access they have to give payment information and verified. Free features are users will be able to see an information dashboard of Dhaka stock exchange and able to see all company lists that are supported by the application. Premium features include a detailed view of the company, historical data plot and the most important one is the stock forecasting. Other than that, the application must be user friendly, show information with live data, and feel good to use. The application must support in both android and ios mobile devices.

2. Stakeholders and Characteristics
2.1 Premium App User
Stakeholder: Individual investors and active traders in Bangladesh interested in in-depth DSE analysis.

Characteristics:
? Tech-savvy and comfortable using mobile apps for financial activities.
? Values detailed company information, historical data analysis, and stock forecasting for informed investment decisions.
? Willing to pay for premium features to gain a competitive edge in the market.
? Expects high accuracy and reliability from stock predictions.
2.2 Free App User
Stakeholder: Individuals interested in staying informed about the DSE but may not require advanced features.
Characteristics:
? May be new to stock trading or have a lower level of investment activity.
? Values a user-friendly app to stay updated on market trends and basic company information.
? Might be interested in upgrading to premium features if they see value in the additional functionalities.
? Data accuracy and live updates are important for informed decision-making.
2.3 External API User
Stakeholder: Financial institutions, investment platforms, or data analysis services.
Characteristics:
? Requires programmatic access to DSE historical and prediction data from BDStockTrend.
? Values reliable data feeds for integration into their own services or applications.
? Interested in secure and well-documented APIs for seamless data access.
? May have specific data format requirements for integration purposes.
2.4 Developer 
To be classified as stakeholders, persons who have a legitimate interest in the product or project must have legitimate interests. Because the developers have too much at stake, they are deemed stakeholders because they have a legitimate interest. As stakeholders, developers are responsible for software delivery and estimation on time.

3. Design and Implementation Constraints
We have employed design and implementation constraints to ensure the success of this project. It also refers to a tool that allows developers and testers to inspect and interact with the application's user interface.
3.1 Mobile Application Development
Development Framework: The mobile application has been developed using Flutter, facilitating a cross-platform solution that supports both Android and iOS devices. This choice ensures a unified codebase, which simplifies maintenance and updates.
Forecasting Capability: The application provides forecasts for over 400 plus companies in Bangladesh, leveraging the state-of-the-art models. This feature is a key value proposition for premium service subscribers.

3.2 Backend Application Development
Technology Stack: The system must utilize a microservices architecture, implemented with technologies that ensure scalability and modular development.
Main Services: A microservice responsible for providing APIs to the mobile application. It also manages scheduled tasks that trigger the data crawling process from the Dhaka Stock Exchange (DSE) and the training/fitting processes for predictive models.
Separate Services for Train/fit: A separate microservice that dedicated to the regular training and updating data of over 400 plus companies. These microservice forecast stock prices for various companies listed on the DSE using the latest market data obtained through scheduled web crawling.
3.3 Performance and Scalability
Handling Computational Load: The architecture must support the computational demands of training over 400 plus models regularly without impacting the performance of the mobile application or the responsiveness of the user interface.
Scalability of Services: Services must be scalable to handle increases in user demand and data volume as the application grows in popularity and scope.

3.4 Security and Compliance
Data Security: Ensuring the security of user data, especially payment information for premium services, is paramount. The system must incorporate robust encryption methods and comply with local data protection regulations.
API Security: APIs provided by the microservices need to implement secure access protocols to prevent unauthorized data access and ensure data integrity.

5. Requirement Engineering Process
Requirements Engineering (RE) determines software requirements according to customer requirements or needs. Requirements engineering process includes requirements elicitation, needs modeling, requirements analysis, requirements assurance & validation, and requirements management.
5.1 Requirement Elicitation Techniques
Requirements elicitation is the practice of researching and finding system requirements for users, customers, and other stakeholders also referred to as "requirement gathering". Requirement elicitation can be done by contacting participants directly or by doing some research, analysis, and testing.
5.1.1 Hold Interviews
We hold discussions that can be held individually or with a small group of participants. They are an effective way to access services without spending a lot of time with participants because we meet with people to discuss only certain important requirements of this program. Negotiations are useful for obtaining individual requirements for members in organizing workshops where those members of the program come together to resolve any issues or conflicts. We mainly perform our interviews based on some specific criteria.
5.1.2 Perform Document Analysis
Existing documentation can help to show how systems are currently operating or what they are what I should do. Documents include written information about current programs, business processes, needs specifications, and competitor research. Review once textual analysis can help determine which performance should remain and functionality that isn’t in use. After the existing document
5.1.3 System Interface Analysis
The first thing to do is to identify which systems the system-to-be shall communicate with. It could be a server on the Internet, a piece of software on the same host as the system-to-be, some hardware, or something completely different.
5.1.4 Distribute Questionnaires
The questionnaire is a useful way to investigate styles, changes in attitudes and users' ideas, and user satisfaction with priorities and preferences. Our lists of questions were as short as possible. The respondent may be tired or frustrated. Had a basic reason for all the questions as well as group the topic areas together for the respondent to focus on. The main advantage of this survey responses was that they were collected in the usual way. Information was summarized by a large number of people.
5.2 Requirement Validation
Requirement validation ensures that the requirements are correct and reflect the quality you want from this program. In the beginning, our requirements looked good but when we read them and tried to work with them, they came out having ambiguities and gaps.
5.2.1 Review the Requirements
Negative peer review, especially the type of rigorous review called evaluation, is unique among the highest quality software processes available. We had a team of reviewers representing different perspectives and carefully examined written needs, analysis models, and related information on disability.
5.2.2 Test the Requirements
The test creates another view of the requirements. We also performed writing tests regarding assurance of whether the expected performance was found or not. Getting tested by the user needs to document the expected product behavior under specified conditions.
5.2.3 Simulate the requirements
To stimulate requirements, trading tools are available that we have used to simulate a proposed system in place or to add details of written requirements. The simulation takes prototyping to the next level.

9.1.1 Three-level Scale 
When a Business Analyst categorizes the requirements in any of the ordering or ranking scale, it is subject to the analyst’s understanding of the business. Many analysts suggest that this method has some drawbacks and advocate methods that have more than one scale. 
9.1.2 Prioritization of the requirements of BDStockTrend
FR-1 - High - Users should be able to register for an account using their email address and a secure password. Login functionality should be secure and allow users to access their accounts

FR-2 - Medium - Users should be able to view a real-time information dashboard of the DSE. The dashboard should display key market indices, gainers and losers, and top traded stocks.

FR-3 - High - Users should be able to browse a comprehensive list of all companies listed on the DSE supported by the application.

FR-4 - Medium - Users with a premium subscription should be able to view detailed information about individual companies, including company profiles, financials, and news.

FR-5 - Medium - Premium users should be able to access historical data plots for stocks, allowing them to analyze past trends.

FR-6 - Medium - The most important premium feature is stock forecasting. This feature should provide users with algorithmic predictions for future stock prices.

FR-7 - Medium - The BDStockTrend system should offer paid APIs for external services to access company historical data and prediction data.

DR-1 - Medium - The application should display live data from the DSE whenever possible. Historical data used for stock analysis and forecasting should be regularly updated.

DR-2 - Medium - Security measures should be implemented to protect user data and ensure data privacy.

PR-1 - High - Application should be with less buffering and load faster, which will increase application usability.

PR-2 - Medium - The application must support the computational demands of training over 400 plus models regularly without impacting the performance of the mobile application or the responsiveness of the user interface.

PR-3 - Medium - All application services must be scalable to handle increases in user demand and data volume as the application grows in popularity and scope.

MR-1 - High - Code must be developed so that it can be modified later and will be readable.

CR-1 - Medium - The application should be compatible with the latest versions of Android and iOS operating systems.
